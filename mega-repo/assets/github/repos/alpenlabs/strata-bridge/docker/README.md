# Bridge Docker Setup

For all images, build from the repository root with:

```sh
docker build -f docker/<Dockerfile> .
```

Layout:

- `docker/base.Dockerfile`: base image used to build the binaries
- `docker/rt.Dockerfile`: shared runtime image
- `docker/strata-bridge/Dockerfile`: bridge operator image
- `docker/secret-service/Dockerfile`: secret-service image
- `docker/asm-runner/Dockerfile`: ASM runner image

## Base image

- x86 ubuntu 24.04 image based on succinct's SP1 image
- SP1 toolchain installed
- Bridge toolchain installed
- External dependencies compiled
- Internal dependencies (`crates` dir) compiled

Build using:

```sh
docker build -f docker/base.Dockerfile . -t bridge-base:latest
```

## Runtime image

- x86 ubuntu 24.04 image updated, upgraded and cleaned

```sh
docker build -f docker/rt.Dockerfile . -t bridge-rt:latest
```

## Running locally

The local Docker stack includes:

- 3 bridge nodes
- 3 secret-service nodes
- 1 ASM runner
- 1 bitcoind regtest node
- 1 FoundationDB instance

Bridge operators can be run locally via containers. These require TLS material, secret-service seeds, bridge params, and a root `.env` file.

### Pre-requisites

To run three bridge operators (one bridge node and one secret service node each), you need to first generate the required TLS certificates:

You can do this with:

```sh
just gen-s2-tls
```

This will create the required files/directories inside `docker/vol/strata-bridge-{1,2,3}`.

You also need seed files for the secret-service nodes at:

- `docker/vol/secret-service-1/seed`
- `docker/vol/secret-service-2/seed`
- `docker/vol/secret-service-3/seed`

Each seed file must contain a random 32-byte value. These will be autogenerated by the secret-service at startup if they don't exist.

### Running the containers

```sh
just clean-docker
```

This will:

1. build `bridge-base` and `bridge-rt`
2. remove persisted service state under `docker/vol/*/data`
3. rebuild the compose images
4. restart the stack from a clean slate

Use:

```sh
just docker
```

to rebuild and restart the stack without deleting persisted state.

### FoundationDB

The bridge nodes use FoundationDB for persistent storage. The Docker workflow automatically:

1. Starts the FoundationDB container first
2. Initializes the database (if not already configured)
3. Waits for it to become healthy
4. Starts the remaining containers

All bridge operators share the same FDB instance but use non-conflicting keyspaces. The initialization is idempotent, so it's safe to run `just clean-docker` multiple times.

The FDB cluster file is pre-configured at `docker/vol/fdb/fdb.cluster` using the Docker service name (`foundationdb:4500`). Data is persisted to `docker/vol/fdb/data`.

If you need to manually initialize or re-initialize FDB, you can run:

```sh
just init-fdb
```

You can check FDB status with:

```sh
docker compose exec -T foundationdb fdbcli --no-status --exec "status minimal"
```

### Bridge Params And Seeds

Bridge params are derived from the secret-service seeds. The three bridge params files should contain the same network-wide covenant key list and protocol values:

- `docker/vol/strata-bridge-1/params.toml`
- `docker/vol/strata-bridge-2/params.toml`
- `docker/vol/strata-bridge-3/params.toml`

Use `dev-cli derive-keys` to generate the key material (musig2, p2p, payout descriptor, wallet
addresses) for each seed without having to run the secret-service/bridge nodes:

```sh
for i in 1 2 3; do
  SEED=$(xxd -p docker/vol/secret-service-$i/seed | tr -d '\n')
  cargo run --bin dev-cli -- derive-keys "$SEED"
done
```

Use that output to populate `docker/vol/strata-bridge-{1,2,3}/params.toml`.

The ASM runner uses the same operator set by reading `docker/vol/strata-bridge-3/params.toml` at startup.

### ASM Runner

The ASM runner is part of the Docker stack and is exposed on:

- `localhost:9010` for JSON-RPC

The relevant ASM files are:

- `docker/vol/asm-runner/config.toml`: checked-in runtime config
- `docker/vol/asm-runner/asm-params-sample.json`: checked-in reference/sample params
- `docker/vol/asm-runner/data`: persisted ASM database

The sample params file is documentation only. It is not used directly at runtime.

At container startup, `docker/asm-runner/entrypoint.sh` runs `docker/asm-runner/gen_asm_params.py`, which:

1. reads `docker/vol/asm-runner/config.toml`
2. reads bridge operator MuSig2 keys from `/bridge/params.toml`
3. queries the live regtest bitcoind node
4. writes an ephemeral params file to `/tmp/asm-params.json`
5. starts `strata-asm-runner --config /app/config.toml --params /tmp/asm-params.json`

Bridge nodes talk to ASM over Docker networking using `http://asm-runner:9010`.

Finally, you need to create a [`.env`](../.env) file at the root of this repository (from where you will run `just clean-docker`).
This file has the following structure:

```plaintext
GENERAL_WALLET_1=bcrt1pjsz8n98943w6h7tn0gtk7rrea6ulp8yhfxsg0gfcmkcl709vavhs5qpe8y
GENERAL_WALLET_2=bcrt1...
GENERAL_WALLET_3=bcrt1...
```

These addresses are funded in the Docker bitcoind startup flow.

### Memory Profiling

The bridge operators and secret services are built with memory profiling enabled using `jemalloc`. This feature exposes HTTP endpoints for heap profiling on port `3000` for each service.

The following ports are mapped for memory profiling:

- Bridge node 1: `localhost:13000` → container port `3000`
- Bridge node 2: `localhost:23000` → container port `3000`
- Bridge node 3: `localhost:33000` → container port `3000`
- Secret service 1: `localhost:11000` → container port `3000`
- Secret service 2: `localhost:21000` → container port `3000`
- Secret service 3: `localhost:31000` → container port `3000`

Available endpoints:

- `/debug/pprof/heap` - Raw heap profile data (`pprof` format)
- `/debug/pprof/heap/flamegraph` - Heap profile as SVG flamegraph

Example usage:

```sh
# Get heap profile for bridge node 1
curl http://localhost:13000/debug/pprof/heap > bridge1-heap.pprof

# View flamegraph for secret service 1 in browser
open http://localhost:11000/debug/pprof/heap/flamegraph
```

### Bridging in

To send a deposit request against the Docker stack, run:

```sh
just bridge-in
```

This uses:

- `target/debug/dev-cli`
- `bin/dev-cli/params.toml`
- `http://localhost:18443/wallet/default`

**Important:** Ensure that the MuSig2 keys in `bin/dev-cli/params.toml` match those used in the Docker bridge params.

### ASM RPC Verification

Useful ASM RPC checks:

```sh
curl -sS -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"strata_asm_getStatus","id":1,"params":[]}' \
  http://localhost:9010/
```

To check deposits, use a recent Bitcoin block hash:

```sh
HEIGHT=$(docker compose exec -T bitcoind bitcoin-cli -regtest -rpcuser=user -rpcpassword=password getblockcount)
HASH=$(docker compose exec -T bitcoind bitcoin-cli -regtest -rpcuser=user -rpcpassword=password getblockhash "$HEIGHT")

curl -sS -H 'Content-Type: application/json' \
  -d "{\"jsonrpc\":\"2.0\",\"method\":\"strata_asm_getDeposits\",\"id\":1,\"params\":[\"$HASH\"]}" \
  http://localhost:9010/
```

This matches the functional-test pattern used in `functional-tests/tests/fn_asm_test.py`.

## Troubleshooting

### FoundationDB Connection Errors

If bridge nodes fail to connect to FoundationDB, check that:

1. The FDB container is running: `docker compose ps foundationdb`
2. The database is initialized: `just init-fdb`
3. The cluster file exists at `docker/vol/fdb/fdb.cluster`

### SP1 Build Errors

If you encounter SP1 verification errors similar to:

```
error: failed to run custom build command for `sp1-prover v5.0.5`
thread 'main' panicked at build.rs:59:47:
called `Result::unwrap()` on an `Err` value: Verification(...vk_map.bin: (verification: FAILED))
```

This typically indicates an issue with the SP1 toolchain setup. Ensure you have:

1. Installed the SP1 toolchain: `curl -L https://sp1up.succinct.xyz | bash`  
2. Completed the setup by running: `sp1up`
3. Restarted your terminal or sourced your shell configuration

If the error persists after proper SP1 setup, verify that your `.env` file contains the correct SP1 configuration as shown above.

### ASM State Mismatch After Restart

If ASM logs warnings about not finding pivot anchor state at genesis, make sure you restarted from a clean slate:

```sh
just clean-docker
```

The ASM database is stored under `docker/vol/asm-runner/data`, so the shared clean recipe must remove that directory when restarting from scratch.
